# RouteSmith v0.3.0 Enterprise Features — Design

**Date:** 2026-06-16
**Previous:** v0.2.0 Stable Release (PRs #23-27, tag v0.2.0)
**Goal:** Add enterprise features: cost model support, provisioned throughput, compliance filtering, per-project isolation, budget enforcement.

---

## Architecture Overview

v0.3.0 adds three new subsystems to the RouteSmith core:

```
┌─────────────────────────────────────────────────────┐
│                 RouteSmith Core (v0.2.0)             │
│  Registry │ Router │ Cache │ Feedback │ Proxy       │
├─────────────────────────────────────────────────────┤
│                 v0.3.0 Additions                     │
│                                                      │
│  ┌─────────────────┐  ┌──────────────────────────┐  │
│  │  Cost Model     │  │  Compliance Filter       │  │
│  │  Subsystem      │  │  Subsystem               │  │
│  │                 │  │                          │  │
│  │ CostModel enum  │  │ compliance_tags on       │  │
│  │ ON_DEMAND       │  │ ModelConfig              │  │
│  │ PROVISIONED     │  │ required_compliance      │  │
│  │ SELF_HOSTED     │  │ per-request param        │  │
│  │ capacity/config │  │ filter pass in registry  │  │
│  └────────┬────────┘  └────────────┬─────────────┘  │
│           │                        │                 │
│  ┌────────┴────────────────────────┴─────────────┐  │
│  │              Strategy Engine                   │  │
│  │                                                │  │
│  │ PROVISIONED_FIRST strategy (new)               │  │
│  │ Overflow routing (provisioned → on-demand)     │  │
│  │ Budget enforcement FAIL/FALLBACK/QUEUE         │  │
│  │ Per-project isolated instances (no shared state)│  │
│  └────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**Key insight:** Compliance filtering happens **first** (hard gate), then **cost model** determines available capacity, then **strategy engine** routes to best model given constraints.

## Design Section 1: CostModel Subsystem

### New Enum

```python
class CostModel(Enum):
    ON_DEMAND = "on_demand"      # Pay per token (current default)
    PROVISIONED = "provisioned"  # Fixed hourly cost, capacity-limited
    SELF_HOSTED = "self_hosted"  # Local/Ollama, near-zero marginal cost
```

### ModelConfig Changes

```python
@dataclass
class ModelConfig:
    # ... existing fields (model_id, cost_per_1k_input, cost_per_1k_output, ...)
    cost_model: CostModel = CostModel.ON_DEMAND

    # Provisioned-specific fields:
    capacity_requests_per_min: int = 0      # Max RPM for provisioned
    provisioned_hourly_cost: float = 0.0    # For utilization reporting only
    provisioned_units: int = 0              # Purchased capacity units
```

### CapacityTracker (new class)

- Per-model counters for provisioned models only
- Tracks: `current_rpm` (rolling 60s window), `total_requests`, `overflow_count`
- Exposed in stats: `provisioned_utilization` (%), `provisioned_overflow` (count)
- Thread-safe with simple locking (not a hot path — routing is <5ms)
- Self-hosted models: treated as cheapest tier, no capacity tracking needed

### Factory Method Updates

All existing factory methods (`with_openai_models()`, `with_groq_models()`, `with_anthropic_models()`) default to `CostModel.ON_DEMAND`. New factory methods for Bedrock provisioned models will set `CostModel.PROVISIONED`.

## Design Section 2: PROVISIONED_FIRST Strategy + Overflow

### Routing Flow

```
Request arrives
    │
    ▼
1. Compliance filter (hard gate)
    │
    ▼
2. Strategy = PROVISIONED_FIRST?
    │
    ├─ YES:
    │   Check provisioned capacity available
    │   ├─ Model has capacity → Use it (cost = $0 marginal)
    │   └─ Model at capacity limit → Overflow to on-demand
    │       │
    │       ▼
    │   Quality prediction on eligible on-demand models
    │   Select best quality/cost among on-demand options
    │
    └─ NO (other strategies):
        Existing routing logic unchanged
```

### Design Details

- `PROVISIONED_FIRST` added to `RoutingStrategy` enum
- Overflow is **built into** PROVISIONED_FIRST, not a separate strategy
- Works with cascade: provisioned → provisioned (if multiple) → on-demand
- Works with parallel: provisioned primary + on-demand backup
- `CapacityTracker` consulted per-request, ~microsecond overhead
- When no on-demand fallback exists: raises `CapacityExhaustedError` (new exception)

### Stats

```python
rs.stats["provisioned_utilization"]  # 0.78 = 78%
rs.stats["provisioned_overflow"]     # 234 overflow requests
rs.stats["by_cost_model"]            # {"on_demand": 234, "provisioned": 4766}
```

## Design Section 3: Compliance Filtering

### ModelConfig Addition

```python
@dataclass  
class ModelConfig:
    # ... existing fields ...
    compliance_tags: set[str] = field(default_factory=set)
    # e.g., {"hipaa", "soc2", "pci", "us-east-1", "eu-west-1"}
```

### Per-Request Filtering

```python
# Pass compliance requirements per-request
rs.completion(
    messages=[...],
    required_compliance={"hipaa"},  # Only models with this tag
)
```

### Registry Method

```python
def filter_by_compliance(self, tags: set[str]) -> list[ModelConfig]:
    """Return models that have ALL required compliance tags."""
```

- Runs BEFORE quality prediction — hard gate, same pattern as capability filtering
- Missing all: raises `NoCompliantModelError` with available tags listed
- Clear error message: `"No models with HIPAA compliance. Available tags: soc2, pci, us-east-1"`

## Design Section 4: Per-Project Cost Allocation (Isolated Instances)

Each project gets its own `RouteSmith` instance with full isolation:

```python
# customer-support project
rs_support = RouteSmith(project="customer-support", config=...)
rs_support.register_model(...)

# internal-search project  
rs_search = RouteSmith(project="internal-search", config=...)
rs_search.register_model(...)
```

The `project` parameter flows into:
- **Logging**: `{project: "customer-support", request_id: "abc123"}`
- **Prometheus metrics**: `routesmith_cost_usd_total{project="customer-support"}`
- **Stats**: `rs.stats["project"] == "customer-support"`
- **Feedback storage**: Separate SQLite file per project (if using default path)

No central server, no shared registry. Aggregation is the user's responsibility.

### RouteSmith __init__ signature change

```python
def __init__(
    self,
    config: RouteSmithConfig | None = None,
    project: str | None = None,  # NEW
):
```

## Design Section 5: Budget Enforcement Behaviors

### New Enum

```python
class BudgetBehavior(Enum):
    FAIL = "fail"          # Raise BudgetExceededError (default)
    FALLBACK = "fallback"  # Route to cheapest eligible model
    QUEUE = "queue"        # Hold until budget window resets (async only)
```

### RouteSmithConfig Addition

```python
@dataclass
class RouteSmithConfig:
    # ... existing fields ...
    budget_behavior: BudgetBehavior = BudgetBehavior.FAIL  # Safe default
```

### Behavior Details

**FAIL** (default):
- `BudgetExceededError` raised before API call
- Error: `"Daily budget of $100.00 exceeded. Current spend: $102.50. Resets in 14 hours."`
- Existing behavior, formalized into enum

**FALLBACK**:
- Router selects **cheapest eligible model** bypassing quality prediction
- Respects compliance + capability filters
- Warning log: `"Budget exceeded, falling back to gpt-4o-mini ($0.00015/1K input)"`
- No eligible model? Falls through to FAIL

**QUEUE**:
- Holds request until budget window resets
- Sync calls raise `BudgetExceededError` with hint to use async
- `rs.acompletion()` returns awaitable that resolves on budget reset
- Use case: batch jobs that can wait overnight

### Stats

```python
rs.stats["budget_events"] = {
    "failures": 3,
    "fallbacks": 47,
    "queued": 0,
    "budget_reset_at": "2026-06-17T00:00:00Z"
}
```

## Implementation Sections

Based on the design, v0.3.0 breaks into 4 implementation sections, dispatched in dependency order:

| Section | Features | Depends On |
|---------|----------|------------|
| **A: CostModel + Capacity** | CostModel enum, ModelConfig fields, CapacityTracker, New exceptions | Nothing |
| **B: Compliance Filtering** | compliance_tags, filter_by_compliance, per-request filtering, NoCompliantModelError | Nothing |
| **C: PROVISIONED_FIRST Strategy** | New strategy, overflow routing, wire into router | A, (B optional) |
| **D: Budget Enforcement + Per-Project** | BudgetBehavior enum, FAIL/FALLBACK/QUEUE, project parameter, isolated stats | Nothing |

Sections A, B, D have no dependencies on each other and can run in parallel. Section C depends on A.
